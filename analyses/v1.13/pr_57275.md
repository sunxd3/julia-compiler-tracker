# PR #57275: Compiler: fix unsoundness of getfield_tfunc on Tuple Types

**Author:** Jameson Nash (vtjnash) | **Merged:** 2025-02-13 | **Risk:** medium

Fixes an inference unsoundness where `getfield_tfunc` could incorrectly constant-fold field access on `Type{Tuple{...}}` (notably `.name`) by assuming tuple types are always `DataType` at runtime. The change makes the analysis conservative for all tuple types and adds a regression test for the `Tuple{Any}` case.

## Changes

`Compiler/src/typeutils.jl:36` (old, removed)
```julia
if t.name === Tuple.name
    return all(isTypeDataType, t.parameters)  # recursively check params
end
return true
```

`Compiler/src/typeutils.jl:36` (new)
```julia
return t.name !== Tuple.name  # all Tuple types are potentially non-DataType
```

`Compiler/test/inline.jl:1748` (new)
```julia
@test !isa(getfield_tfunc(Type{Tuple{Any}}, Core.Const(:name)), Core.Const)
```

## Breaking for downstream tools

None

## Behavioral differences

`getfield_tfunc` no longer returns a `Core.Const` for `.name` on `Type{Tuple{...}}` (including `Type{Tuple{Any}}`), avoiding incorrect constant-folding when the runtime type could be a `Union` due to tuple covariance.
